## 시스템 프로그래밍의 이해와 접근, 컴퓨터 하드웨어의 구성/CPU에 대한 이해

### 시스템(컴퓨터 시스템)의 범위

=> 하드웨어 + 운영체제
ex) CPU가 intel기반이고 운영체제는 windows다.

### 시스템 프로그래밍

* 컴퓨터 시스템을 활용하는 소프트웨어 개발
* windows 운영체제의 기능을 십분 활용하는 프로그래밍

### 응용 소프트웨어 개발과의 차이점

* 시스템 프로그래밍은 모든 응용 프로그램에 포함되는 요소

### 컴퓨터 시스템의 주요 구성요소

* CPU, 캐쉬
  * 컴퓨터 하드웨어 구조
* 운영체제
  * 메인 메모리
* 하드디스크 
  * 파일I/O (다양한 I/O포함)

### 컴퓨터 하드웨어 구성

* CPU ( Central Processing Unit)
  * 중앙 처리 장치
  * 연산이 이루어지는 원리는 무엇인가? => 연산은 프로그램 실행과정의 일부분이다.

* 메인 메모리 
  * 램(RAM)
  * 프로그램 실행 방식을 이해하는 것

* 입출력 버스
  * 데이터 송수신이 이루어지는 원리 

### CPU에 대한 이해(전체 구성)

* ALU(Arithmetic Logic Unit)
  * 연산 담당 

* 컨트롤 유닛
  * CPU를 총괄하여 신호를 주는, 가장 많은 컨트롤을 하는 요소
  * 명령어 해석, CPU 행동 결정 
  
* 버스 인터페이스
  * 버스가 어덯게 데이터를 주고 받는지 알고 있다.
  * 들어온 데이터를 레지스터에 저장 

### 클럭 신호(Clock Pulse)

* 동작 타이밍
  * 클럭 발생기의 클럭

* 필요성
  * 요소들의 동기화

* 시스템을 안정적으로, 데이터를 분실하는 것을 막기 위해 클럭 신호를 제공한다.

## 프로그램의 실행과정, 하드웨어 구성의 재접근

### 프로그램 실행 과정

전처리기 -> 컴파일러 -> 어셈블러 -> 링커

* 전처리기에 의한 치환 작업
  * #으로 시작하는 지시자

* 컴파일러에 의한 번역
  * CPU의 명령어로 번역

* 어셈블러에 의한 바이너리 코드 생성
  * CPU의 명령어를 바이너리 코드로 번역

* 링커에 의한 연결과 결합
  * 라이브러리와의 결합
  * 실행 파일을 만드는 것


### Stored Program Concept

* 명령어는 메모리에 저장되어서 cpu에 의해 1.fetch되고 2.decode되고 3.execution 되어야 한다.

* Fetch
  * CPU 내부로 명령어 이동(버스 인터페이스를 통해서) 

* Decode
  * 명령어 해석
  * 컨트롤 유닛에 의해 해석

* Execution
  * 연산을 진행
  * 보통은 ALU를 생각 => 연산이라는 건 CPU가 하는 궁극적인 일, Execution이 더 큰 의미
  * => ALU가 중심이 되어서 요소들이 협력하여 execution 된다.

* 명령어가 메모리에 저장되있는 상태에서 명령어를 cpu로 하나씩 가져와서 명령어가 어떤 의미를 가지는지 decode(해석)하고 execution(실행)을 시킨다.


## Windows에서의 유니코드(UNICODE)

### 문자셋의 종류와 특성

* SBCS(Single Byte Character Set)
  * 문자를 표현하는데 1byte 사용
  * 아스키 코드

* MBSC(Multi Byte Character Set)
  * 한글은 2byte , 영문은 1byte 사용 => 효율적이라고 생각할 수 있지만, 발전하면서 문자표현으로는 부담X

* WBCS(Wide Byte Character Set)
  * 문자를 표현하는데 2byte 사용 => 안정적, 메모리 차이 별로 없음
  * 유니코드 


## 프로세스와 스케줄러의 이해

### 프로세스

* 메인 메모리로 이동하여 실행 중인 프로그램 -> 일반적인 정의

### 프로세스의 범위

* 메모리 구조 + 레지스터 Set
* 프로세스 별 독립적인 대상은 프로세스의 범주에 포함시킬 수 있다.
=> 프로세스를 구성하는 범주는 프로세스 별로 독립적으로 할당받는 리소스다.

### 프로세스 스케줄러

* 스케줄러도 하나의 소프트웨어다.
  * 스케줄러가 동작하는 순간에 프로세스들은 동작할 수 없다.
=> 넓은 관전에서 스케줄러도 하나의 프로세스다.

* 프로세스 스케줄러 기능
  * 둘 이상의 프로세스가 적절히 실행되도록 컨트롤

* 스케줄링 방법
  * 스케줄링 알고리즘에 따라 다양함.


### 프로세스 상태

* 프로세스는 상태 정보를 가진다.
  * Running - cpu에 의해 실행중인 상태, cpu에 코어가 하나면 running 상태의 프로세스는 maximum 하나다.
  * Ready - 프로세스는 모든 준비가 끝나고, Running 하고 싶은 상태, 스케줄러가 선택해 주기를 기다리는 상태
  * Blocked - cpu의 도움을 크게 요구하지 않는 연산의 경우 blocked 상태가 됨 -> ex) I/O연산, I/O연산이 끝나면 다시 Ready상태가 된다. 

=> 프로세스 a,b,c가 실행 중일 때 마치 동시에 실행되는 것처럼 보임 -> cpu가 고속으로 번갈아 가면서 실행 중인것.
=> Running과 Ready 상태를 왔다 갔다 함


## 커널 오브젝트에 대한 이해

### 커널 오브젝트

* 커널 - 운영체제가 가지고 있는 핵심 부분
* 커널에 의해 관리되는 리소스(운영체제에 의해 생성,소멸되는 것) 정보를 담고 있는 데이터 블록
* 운영체제에 의해 관리되어지는 리소스들은 운영체제가 관리하기 위해 데이터 블록을 내부적으로 저장해둔다. => 커널 오브젝트
* 운영체제에 의해 관리되어지는 리소스들 중에 커널오브젝트의 생성을 동반하지 않는 것이 없다.

### 커널 오브젝트와 핸들의 관계

* 커널 오브젝트는 OS가 관리, 프로그래머가 직접 접근할 수 없다.
* 운영체제는 커널 오브젝트가 생성될 때 고유한 번호를 매김 -> 핸들 정보
* 핸들을 통해 간접적으로 접근할 수 있다.


## 프로세스의 스케줄링

### 일반 OS와 리얼타임 OS

=> 스케줄러가 어떻게 스케줄링 하느냐에 따라 일반OS와 리얼타임OS가 결정이 된다.

* 리얼타임 OS - 실시간, 바로바로 응답한다.
  ex) A프로세스를 실행 시켰을 때,
      시작은 리얼타임os가 빠름, 다만 계속 빠른 것은 아니다.
      실행속도는 cpu관점, 실행 속도는 cpu에 의존적이다.
      
  Q. 시작 속도가 빠르다는 것은 cpu가 빠르다는 것이 아닌가?
  -> a,b,c 프로세스가 있을 때 클럭을 기준으로 타임슬라이스를 정한다.
     예를들어 1클럭이 1초, 타임슬라이스를 3초라고 했을 때, 타임슬라이스는 3클럭
     일반 os는 타임슬라이스가 길다.
     리얼타임 os는 타임슬라이스가 작아서 응답성이 좋다.
     
### 선점형 OS와 비선점형 OS

* 선점형 os
  * 스케줄러가 일을 더 많이 한다, 스케줄러가 프로세스 실행에 직접적으로 관여하는 것.
  * a프로세스 우선순위 2, b프로세스 우선순위가 3 (더 높은 우선순위) 이면
  * a프로세스가 실행 중일 때 우선순위가 높은 b프로세스가 생성되면 스케줄러가 a프로세스를 밀어내고(할당된 타임슬라이스를 소모 후), b프로세스 실행

* 비선점형 os
  * 스케줄러가 하는 일이 적다, 스케줄러가 하는 일이 선점형 os보다 제한적이다.
  * 우선순위가 높은 프로세스가 생성되도 바로 양보하지 않고 실행중인 프로세스가 명시적으로 선언해야지만 우선순위가 높은 프로세스가 실행될 수 있음.
